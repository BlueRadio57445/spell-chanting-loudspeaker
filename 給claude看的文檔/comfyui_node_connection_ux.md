# ComfyUI 節點連線互動設計：UX/UI 完整指南

> 本文件描述 ComfyUI 節點資料流的所有使用者互動細節，涵蓋從最基礎的連線規則到最細微的視覺回饋。這些細節彼此環環相扣，共同構成讓使用者感到「自然、流暢、直覺」的操作體驗。任何一個細節的缺失都可能讓使用者在某個時刻感到困惑或受阻。

---

## 一、最核心的非對稱規則：輸出一對多，輸入一對一

理解 ComfyUI 節點連線最重要的前提，是理解「節點就是函式」這個隱喻。

一個函式可以把計算結果**同時交給很多地方使用**，但每一個參數只能接受**一個來源的值**。ComfyUI 完整實踐了這個語意：

- **輸出端口（Output）** 儲存的是一個 Link ID 的**陣列**（`output.links: LinkId[]`），可以無限增長。同一個輸出可以同時連接到十個、二十個不同的輸入，資料會被平行地傳遞給所有下游節點。
- **輸入端口（Input）** 儲存的是**單一一個** Link ID（`input.link: LinkId | null`），永遠只能接受一條連線。

這個非對稱設計讓整個資料流圖保持了函式語意的一致性：資料從輸出「廣播」出去，但每個輸入端口的值只有一個確定的來源，不存在「多個來源競爭同一個輸入」的模糊狀態。

### 舊連線的自動替換

當使用者把一條新連線拖到**已經有連線的輸入端口**時，系統不會拒絕，也不會跳出確認對話框。舊的連線會被**靜默地自動斷開**，新的連線直接接上。

```
輸入端口已連線 A → 使用者拖入 B → A 的連線斷開，B 的連線建立
```

這個設計的用意是「最後一個操作最重要」——使用者既然決定拖新的連線過來，就代表他想要的是新的來源，系統不應該在這裡設置障礙。這是一個「寬容優先」的設計決策。

### EVENT 類型的特殊限制

有一個例外：當輸出端口的類型是 `EVENT`（事件型訊號，值為 `-1`），且系統設定 `allow_multi_output_for_events` 為 `false`（預設值）時，EVENT 輸出端在建立新連線前會**先斷開所有既有連線**。這是因為事件型訊號的語意通常是「只有一個觸發目標」，允許多重連線可能造成語意混亂。

## 二、建立新連線的完整流程

### 從輸出端口拖曳（Output → Input）

資料向右流動的設計讓起點是節點右側的輸出端口。使用者在輸出端口上按下左鍵並開始拖曳時，系統進入連線模式（`state.connectingTo = 'input'`），一條**貝茲曲線橡皮筋線**從端口位置延伸出來，跟著滑鼠游標移動。

這條橡皮筋線的彎曲方向有語意：從輸出端出發的線條**向右彎曲**，視覺上表達「我正在向右尋找目的地」。線條的起點固定在輸出端口圓點的位置，終點跟著游標移動，讓使用者始終清楚「連線的來源」在哪裡。

### 從輸入端口拖曳（Input → Output）

使用者也可以從**空的輸入端口**開始拖曳。這時系統進入反向連線模式（`state.connectingTo = 'output'`），橡皮筋線**向左彎曲**，語意變成「我正在向左尋找資料來源」。

這個雙向起點的設計非常重要：使用者在不確定「應該從哪一邊開始拖」的情況下，兩側都能作為起點。很多初學者直覺地從輸入端開始拖，系統同樣支援，不會因為操作方向不同而拒絕。

### 放開滑鼠的四種情境與結果

**情境一：放在目標節點的端口上（精準對準）**

系統在每次 mousemove 都計算端口的 20×20 像素點擊框，使用者只要在這個範圍內放開，連線就建立完成。

**情境二：放在目標節點的空白區域上（未精準對準端口）**

這是一個對使用者非常友善的設計：使用者不需要精準瞄準端口圓點，**把連線放在節點的任何位置**，系統會自動在該節點上尋找第一個類型相容的端口完成連線。這讓連線操作的容錯率大幅提升，使用者不需要像外科醫生一樣精準地點擊一個小圓點。

這個自動尋找的邏輯（`findInputByType` / `findOutputByType`）按照端口順序逐一掃描，取第一個類型相容且可用的端口。

**情境三：放在 Reroute（分路點）上**

連線與分路點相連，資料路徑在此轉折或分叉（詳見第八節）。

**情境四：放在空白畫布上（未命中任何節點）**

連線建立失敗，橡皮筋線消失，畫面恢復到拖曳前的狀態，不留任何痕跡。此時若系統設定 `release_link_on_empty_shows_menu` 為 `true`，且使用者同時按住 Shift 鍵，則會開啟搜尋框，讓使用者直接搜尋並新增節點。

## 三、類型相容性：連線的守門員

### 即時視覺回饋

當使用者正在拖曳橡皮筋線時，系統在每一個 mousemove 事件都持續判斷游標下方的端口是否相容。相容的端口會高亮顯示；不相容的端口暗淡，不提供接受的視覺邀請。使用者在移動過程中就能預知哪裡可以放、哪裡不行，而不是等到放開才得到反饋。

### 類型比對的五個層次

**層次一：萬用字元（Wildcard）**
若任一端的類型是 `"*"` 或空字串，則視為通用端口，可以連接任何類型的資料。這讓「轉接節點」、「路由節點」等通用工具能夠靈活適應各種資料流，不被類型限制。

**層次二：類型相等**
`IMAGE` 連 `IMAGE`、`LATENT` 連 `LATENT`，最常見的情況，字串直接比對（不分大小寫）。

**層次三：多類型（Comma-separated）**
端口可以宣告接受多種類型，例如 `"INT,FLOAT"`。只要兩端的類型集合有**任何一對組合相容**，連線就被允許。例如輸出是 `"INT,FLOAT"`，輸入是 `"FLOAT,STRING"`，兩者有 `FLOAT` 交集，連線成立，連線本身被標記為 `FLOAT` 類型（最精確的共同類型）。

**層次四：EVENT ↔ ACTION**
這兩個特殊類型（值均為 `-1`）可以互相連線，代表控制流訊號，不帶資料內容，只傳遞「觸發」的意圖。

**層次五：節點自訂攔截**
每個節點可以實作 `onConnectInput()` 和 `onConnectOutput()` 回呼，在類型檢查通過後進一步決定是否允許連線。回呼若回傳 `false`，連線被靜默拒絕。這讓節點能做更細緻的業務邏輯判斷，例如「只有在某個條件成立時才接受連線」。

### 連線失敗是靜默的

當類型不相容，`connectSlots()` 回傳 `null`，連線不建立，畫面重繪（`setDirtyCanvas`），但**沒有錯誤提示視窗**。系統的策略是：在使用者拖曳移動期間就已經透過視覺高亮告知哪裡可以放，到了放開的那一刻，使用者應該已經知道結果了。

### 後端的雙重驗證（執行時）

前端的即時類型檢查是第一道防線。當使用者送出整個工作流程執行時，後端（Python 端）會再做一次嚴格驗證：

- 確認連線格式正確（每條連線是 `[節點ID, 輸出槽索引]` 的二元列表）
- 確認來源節點的 `RETURN_TYPES` 與目標輸入的型別相容
- 支援 strict 模式（received types 必須是 expected types 的子集）和寬鬆模式（只要有交集）
- 若後端驗證失敗，回傳詳細錯誤：哪個節點、哪個端口、收到什麼類型、期望什麼類型

## 四、重新路由已建立的連線

### 移動輸入端的連線

一條已建立的連線，使用者想把它的**終點（輸入端）換到別的地方**，有兩種觸發方式：

- 按住 **Shift** 再點擊已連線的輸入端口
- 若系統設定開啟 `allow_reconnect_links`，則直接點擊已連線的輸入端口即可

觸發後，這條連線進入「移動模式」（`MovingInputLink`）：**起點（輸出端）保持完全不動**，終點變成跟著游標移動的橡皮筋線。原本的連線在視覺上消失（`link._dragging = true`），讓使用者看到的是「這條線被我拿起來了，準備放到別的地方」的狀態。

#### 「安全放開區」圓圈的設計

這是一個細心的安全機制。在原本輸入端口的位置，會出現一個**半徑 35 像素的圓圈**。這個圓圈代表「退場緩衝區」，視覺上暗示使用者「這裡是原本的位置」。

- 滑鼠在圓圈**外面**放開到空白處 → 連線斷開（原本的連線消失）
- 滑鼠在圓圈**裡面**放開到空白處 → 連線**維持原狀**，什麼都不會改變

這個圓圈讓「不小心點到了」或「拿起來又不確定要放哪」的情況可以安全地撤回，使用者不需要 Ctrl+Z 就能「取消」這次操作。

### 移動輸出端的連線

按住 **Shift** 點擊已連線的輸出端口，系統會把**所有**從該輸出端口出發的連線**全部一起移動**（`state.multi = true`）。如果一個輸出端連了三條線到三個不同節點，Shift+點擊後，這三條線會同時變成橡皮筋，可以一次重定向到新的輸出端口。

這對於「我想換這個節點用的模型來源」這類操作非常有用——一個動作就能重定向所有下游連線。

## 五、連線的視覺樣式

### 連線的類型決定顏色

每條連線的顏色由它傳遞的資料類型決定，讓使用者透過顏色就能快速分辨資料流的內容：

- **EVENT 類型**（控制訊號）→ `#A86`（橘棕色）
- **number 類型** → `#AAA`（灰色）
- **node 類型** → `#DCA`（棕黃色）
- 其他類型由節點定義自訂顏色，或繼承預設樣式

端口自身也有兩個顏色狀態：`color_off`（未連線時）和 `color_on`（已連線時），讓使用者可以直接從端口顏色看出連線狀態。

### 連線的樣式類型

系統支援三種連線渲染方式，可透過設定切換：

- **Spline（貝茲曲線）**：預設，圓滑的 S 形曲線，視覺最自然
- **Linear（折線）**：直線折角連接，適合需要精確空間排列的場景
- **Hidden（隱藏）**：連線不顯示，適合截圖或展示用

### 連線的中點標記

連線在中點位置可以顯示標記（marker），有三種模式：

- **None**：無標記（拖曳中的橡皮筋線不顯示標記）
- **Circle**：圓點，表示資料流方向的中繼點
- **Arrow**：箭頭，明確指示資料流向

### 端口的形狀語言

每個端口可以設定不同形狀（`RenderShape`），形狀本身傳遞語意：

- `BOX`（方形）：一般資料端口
- `CIRCLE`（圓形）：常見的預設形狀
- `ARROW`（箭頭）：有方向性的訊號端口
- `HollowCircle`（空心圓）：通常表示「尚未連線」的待接受狀態
- `GRID`、`CARD`：特殊用途端口

### 連線懸停（Hover）的視覺放大

當使用者的游標移動到一條已建立的連線上時，系統透過比實際線寬**更寬的隱形碰撞區域**（`connections_width + 7` 像素）來偵測懸停，讓使用者更容易「命中」一條細細的連線。命中後，連線會以高亮色和更粗的線條顯示，並在中點顯示標記，告知使用者「這條線現在是焦點」。

### 錯誤狀態標記

端口有一個 `hasErrors` 屬性，可以在驗證失敗時以視覺方式標記（通常是不同顏色或圖示），讓使用者一眼看出哪個端口有問題。

## 六、斷開連線的四種方式

### 方式一：Alt + 點擊輸入端口（最快速、最直接）

在已連線的輸入端口上按住 **Alt** 鍵再點擊，連線**立即斷開**，沒有確認步驟，沒有動畫，執行完就完成了。這是為熟練使用者設計的快捷路徑，一個動作完成整個斷線操作。

### 方式二：把連線拖離並放開在空白處

使用 Shift+點擊（或開啟 `allow_reconnect_links` 後直接點擊）把連線的輸入端拖起來，然後把游標拖到遠離原始位置的空白畫布上放開（超出 35px 圓圈範圍），連線斷開。

這個方式的設計意圖是讓「重新路由」和「斷開」共享同一個手勢——拖曳的動作可以在中途改變心意：中途發現更好的目的地就放上去，沒有找到合適的就放開到空白處斷線。操作流暢，不需要切換模式。

### 方式三：在安全圓圈內放開（撤回拖曳）

把連線拖起來後，在 35px 圓圈範圍內放開，維持原狀。這不完全是「斷線」，更像是「取消這次移動操作」。詳見第四節。

### 方式四：右鍵選單（最明確、最可探索）

對節點或連線按下右鍵，彈出上下文選單，提供：

- **Remove link** — 斷開特定一條連線
- **Disconnect Input X** — 斷開某個輸入端口的連線
- **Disconnect All** — 斷開節點所有連線

右鍵選單是最「可探索」的操作方式——使用者不需要記住鍵盤快捷鍵，只需要右鍵，在選單裡找到對應的選項。適合初學者或不常用這個操作的情境。

## 七、斷線後的完整資料清理

斷開連線不只是視覺上的消失，系統會進行完整的「清場」確保資料圖的一致性：

1. **從來源節點的輸出陣列中移除**：`output.links` 陣列中的對應 LinkId 被刪除（`splice`）
2. **清空目標節點的輸入記錄**：`input.link` 設為 `null`
3. **自動清理孤立的 Reroute**：連線路徑上若有 Reroute 分路點，且斷線後該 Reroute 上沒有任何剩餘的連線通過，Reroute 點會**自動從圖中移除**，不會留下視覺垃圾。（可透過 `keepReroutes=true` 選項保留 Reroute）
4. **觸發節點回呼**：來源節點和目標節點各自的 `onConnectionsChange()` 回呼被觸發，讓節點有機會更新自身的顯示狀態（例如重新計算尺寸、顯示佔位符等）
5. **圖版本號遞增**：`graph._version++`，讓外部系統知道圖結構發生了變化
6. **畫布標記重繪**：前景層（`fg=true`）標記為需要重繪，下一幀更新畫面

這套清理機制確保了視覺狀態和資料狀態永遠同步——使用者看到的連線就是真正存在的連線，看不到的就是已經斷開的。

## 八、Reroute 分路點：整理線路的工具

Reroute 是可以放在連線中間的視覺轉折點，讓複雜的連線路徑得以整理，讓線路繞過節點或以更清晰的方式呈現。它有幾個重要的行為細節。

### 外觀

Reroute 的視覺直徑是 10px（半徑 5px），以圓形表示。預設填充色是深藍色（`#18184d`），但顏色會隨其所連接的連線類型變化。若 Reroute 被選中，外圍會出現一個 1.2 倍半徑的白色高亮圓環。若 Reroute 只連接了浮動連線（拖曳中的臨時連線），透明度會降到 33%，表示這是一個「不完整」的狀態。

### 懸停區域比視覺範圍更大

Reroute 本身很小（10px 直徑），但懸停的感應範圍比視覺更寬廣。當游標進入懸停區域但尚未進入 Reroute 圓圈時，兩側的輸入/輸出端點（小圓點，半徑 5px）會以灰色輪廓顯示，提示使用者「這裡有可操作的連接點」。當游標進入 Reroute 圓圈本身時，最近的端點填充為連線顏色，表示「你現在可以從這個點拖曳」。

### 點擊連線線段插入 Reroute

當使用者點擊已建立連線的**線段中間部分**，系統在點擊位置插入一個 Reroute 點，原本的一段連線分成兩段，Reroute 成為中間的轉接點。插入後，Reroute 立即進入拖曳狀態，使用者可以直接拖曳來調整位置，不需要額外的點擊。

### 從 Reroute 點拖出新連線

點擊 Reroute 點的輸出端（右側點）並拖曳，可以從此分路點引出新的連線分支，實現「一個輸出連接到多個地方，中途經過分路點整理路徑」的視覺效果。

### Alt + 點擊刪除 Reroute

在 Reroute 點上按住 Alt 並點擊，刪除這個分路點。若 Reroute 刪除後前後仍有有效連線，系統嘗試重新合併；若合併後的連線已無意義（例如浮動連線），則整段路徑被清除。

### Reroute 的鏈式結構

多個 Reroute 可以形成鏈（通過 `parentId` 追蹤），資料從來源流經每個 Reroute 點到達目的地。任何一個 Reroute 點上的連線完全斷開後，該點從鏈中移除，前後的 Reroute 自動重新連接成新的鏈。

### 對齊網格

Reroute 點支援 Snap-to-Grid，拖曳時會按照網格間距對齊（`snapToGrid(snapTo)`），保持畫面整齊。

## 九、拖曳的邊界體驗：自動捲動

當使用者拖曳連線的游標移向畫布邊緣時，畫布會**自動開始捲動（Auto-Pan）**。速度與游標靠近邊緣的程度成比例，讓使用者可以不間斷地拖到畫布外的遠端節點，而不需要先放開連線、捲動畫面、再重新開始拖曳。

捲動過程中，`graph_mouse` 座標根據畫布偏移持續更新，橡皮筋線的終點始終對應真實的游標位置，使用者看到的是連續流暢的拖曳，而不是跳動或錯位。

放開滑鼠或連線完成的瞬間，自動捲動立即停止（`_autoPan.stop()`），不會有慣性繼續移動的干擾。

## 十、點擊與拖曳的辨別：容錯的關鍵

系統透過 `CanvasPointer` 類別精確辨別使用者的意圖是「點擊」還是「拖曳」，有兩個判斷條件（任一成立就認定為拖曳）：

1. **時間閾值**：按下滑鼠後超過 **150 毫秒**才放開，視為拖曳（靜止點擊的人不會等那麼久）
2. **位移閾值**：滑鼠移動超過 **6 像素**，視為拖曳（手抖 6px 以內不算拖曳）

在這個閾值以內的所有操作都被判定為「點擊」——使用者即使手抖了一點點，也不會意外地把一個端口點擊觸發成連線拖曳。

### 雙擊的辨別

系統同樣追蹤雙擊：連續兩次按下的時間間隔在 `CanvasPointer.doubleClickTime`（300ms）以內，且兩次點擊位置接近，判定為雙擊，觸發 `onDoubleClick` 回呼。

- **雙擊空白畫布** → 開啟搜尋框（若 `allow_searchbox` 啟用）
- **雙擊子圖節點的標題欄** → 進入子圖（`openSubgraph()`）
- **雙擊一般節點** → 觸發節點自訂的 `onDblClick` 回呼

### 按下就捕獲指針

當按下滑鼠的瞬間，系統呼叫 `element.setPointerCapture(pointerId)`，確保即使游標移出瀏覽器視窗，拖曳事件仍然被正確接收。放開時才呼叫 `releasePointerCapture`。

### 多指觸控防護

在觸控設備上，當系統偵測到多指觸控（捏合縮放手勢），會暫停所有連線拖曳邏輯（`processMouseDown` 提前返回），避免縮放手勢和連線操作互相干擾。只有單指操作才進行連線互動。

## 十三、中鍵點擊端口：智慧型節點建議

在端口上按下**滑鼠中鍵**（若系統設定 `middle_click_slot_add_default_node = true`），系統根據該端口的類型，在 `registered_slot_out_types`（輸出類型對應的預設節點列表）或 `registered_slot_in_types` 中查詢，在游標附近展示建議清單，選擇後**自動新增節點並連線**。

這個功能省去了「手動新增節點 → 手動連線」兩個步驟，在「我知道要往哪個方向走，但還沒決定具體節點」的情境下特別有價值。

## 十四、Widget（控制元件）與連線的交互

節點的輸入端口可以附帶一個 Widget（滑桿、下拉選單、文字框等），讓使用者直接在節點上設定固定值，不需要連線。當這個輸入端口接受到一條外部連線時，Widget 會**自動隱藏**，被連線值取代——意思是「這個值現在由外部決定，不需要在這裡手動設定」。

若輸入端口設定了 `alwaysVisible = true`，即使有連線接入，Widget 仍然顯示（用於某些需要同時顯示連線值和手動值的特殊情境）。

### Widget 的拖曳互動

Widget 本身也可以有 `onPointerDown`、`onDrag` 等事件。系統在處理節點點擊時，會先嘗試 `widget.onPointerDown(pointer, node, canvas)`；若回傳 `true`，表示 Widget 自己處理了這個事件，系統不繼續向下傳遞。這個設計讓滑桿、旋鈕等需要拖曳操作的 Widget 能夠在節點內部正常運作，不會與連線拖曳衝突。

---

## 十五、節點的折疊與縮放

### 折疊

節點**左上角**有一個固定大小的折疊按鈕（`NODE_TITLE_HEIGHT × NODE_TITLE_HEIGHT` 像素的方形區域）。點擊後節點縮小到只顯示標題欄，所有端口和 Widget 隱藏，但連線仍然保持有效。折疊的節點用 `flags.collapsed = true` 標記，顯示寬度縮小到 `_collapsed_width`。

折疊是一個常用的整理工具，讓複雜的工作流程可以把暫時不需要調整的節點「收起來」，騰出畫面空間。

### 縮放（Resize）

每個節點的四個角落都有 `LGraphNode.resizeHandleSize` 像素的縮放控制點（`CompassCorners`：NE、SE、SW、NW）。拖曳任一角落可以調整節點大小，縮放支援網格對齊。

節點可以設定 `resizable = false` 禁用縮放。縮放完成後觸發 `onResize(size)` 回呼。

## 十六、子圖（Subgraph）的特殊行為

ComfyUI 支援將一組節點打包成「子圖節點」（SubgraphNode），子圖有自己的輸入節點（SubgraphInputNode）和輸出節點（SubgraphOutputNode）。

- **雙擊子圖節點標題** → 進入子圖（`openSubgraph(node.subgraph, node)`），畫布切換到子圖內部視圖
- **子圖的 IO 節點** 在連線拖曳的落點判斷中有最高優先級（先於一般節點判斷），確保跨子圖邊界的連線操作正確路由
- 拖曳連線到子圖 IO 節點上，呼叫 `dropOnIoNode()` 而非 `dropOnNode()`，走專屬的連線邏輯

## 十七、連線物件的完整資料

每一條存在的連線（`LLink`）攜帶以下資料，理解這些資料有助於理解為什麼連線系統能做到上述所有行為：

| 欄位 | 意義 |
|------|------|
| `id` | 唯一連線 ID，從 `graph.state.lastLinkId` 遞增 |
| `type` | 連線傳遞的資料類型（兩端的共同類型） |
| `origin_id` | 來源節點 ID |
| `origin_slot` | 來源節點的輸出槽索引 |
| `target_id` | 目標節點 ID |
| `target_slot` | 目標節點的輸入槽索引 |
| `parentId` | 若有 Reroute，指向第一個 Reroute 的 ID |
| `_pos` | 連線曲線的中點座標（每幀計算，用於中點標記渲染） |
| `path` | 連線的 `Path2D` 物件（用於懸停碰撞偵測） |
| `_dragging` | 為 `true` 時代表正在被拖曳，連線視覺上隱藏 |

「浮動連線」（Floating Link，任一端 ID 為 `-1`）是拖曳過程中的臨時物件，儲存在 `graph.floatingLinksInternal`，拖曳完成後轉換為正式連線或被丟棄。

## 十八、圖的完整性保護機制

### 重複連線清除

`graph._removeDuplicateLinks()` 按照（來源節點、來源槽、目標節點、目標槽）四元組分組所有連線，若發現完全相同的重複連線，只保留一條，其餘刪除。這確保了資料圖的語意唯一性。

### 節點刪除時的連線清理

當一個節點被從圖中移除時，系統自動清理所有與之相連的連線：

- 遍歷節點所有輸入槽，對每個有連線的槽呼叫 `disconnectInput()`
- 遍歷節點所有輸出槽，對每個有連線的槽呼叫 `disconnectOutput()`
- 清理引用該節點的浮動連線

節點刪除後，不會留下任何「指向已刪除節點」的殘留連線。

### 事件通知鏈

每次連線建立或斷開：

1. `graph.beforeChange()` → 觸發 `onBeforeChange`、`canvas.onBeforeChange`（用於 Undo/Redo 記錄）
2. 連線操作執行
3. 相關節點的 `onConnectionsChange(type, slot, connected, link, slotInfo)` 被呼叫
4. `graph.onConnectionChange(node)` 被呼叫（圖層級的通知）
5. `graph.afterChange()` → 觸發 `onAfterChange`、`canvas.onAfterChange`

這個完整的事件通知鏈確保了 Undo/Redo 系統、外部擴充套件、以及節點自身的業務邏輯都能對連線變化做出反應。

## 十九、LinkConnector 的事件系統（擴充套件接口）

`LinkConnector` 透過 `CustomEventTarget` 暴露一套可以被外部擴充套件監聽和攔截的事件：

| 事件名 | 時機 | 可攔截？ |
|--------|------|---------|
| `before-drop-links` | 放開前 | ✅ 回傳 `false` 可取消整個放開操作 |
| `after-drop-links` | 放開後（無論成功與否） | ❌ |
| `dropped-on-node` | 放在節點上但未命中端口 | ✅ 回傳 `false` 取消自動對齊連線 |
| `link-created` | 新連線建立成功 | ❌ |
| `input-moved` | 輸入連線重新路由成功 | ❌ |
| `before-move-input` | 即將開始移動輸入連線 | ✅ 回傳 `false` 阻止移動 |
| `before-move-output` | 即將開始移動輸出連線 | ✅ 回傳 `false` 阻止移動（逐條） |
| `reset` | LinkConnector 即將重置 | ✅ 非強制重置時可阻止（force=true 不可攔截） |

這套事件系統讓擴充套件能夠在不修改核心程式碼的情況下，加入自訂的連線行為（例如：特定類型的連線建立時自動插入一個轉換節點）。

## 二十、整體設計哲學

ComfyUI 節點連線的 UX 設計圍繞著幾個貫穿始終的核心原則：

**即時反饋，不留懸念**
使用者的每一個動作都在下一個畫面幀得到視覺回應。系統不等操作完成才告知結果，而是在過程中就持續透過高亮、線條彎曲方向、游標形狀等方式預告可能性。

**寬容第一，不懲罰誤操作**
150ms 時間閾值、6px 位移閾值、35px 安全圓圈——這些數字都在說同一件事：系統假設使用者是有意圖的，不會因為輕微的手抖或短暫的猶豫而觸發不必要的操作。

**方向一致，只需學一次**
「右側輸出、左側輸入、資料向右流、線條彎曲方向即操作意圖」——這套視覺語言貫穿所有節點、所有類型、所有操作，使用者建立一次心智模型後，在任何情境下都能適用。

**多進路，尊重不同習慣**
同一個目標有多種達成方式：可以從輸出端拖，也可以從輸入端拖；可以 Alt+點擊斷線，也可以拖曳再放開，也可以右鍵選單。不同習慣的使用者都能找到符合自己直覺的那條路。

**狀態透明，可以信任眼睛所見**
視覺狀態永遠和資料狀態同步。沒有「看起來連了但實際沒連」，沒有「視覺消失了但資料還在流動」。使用者可以完全信任自己在畫面上看到的圖，不需要透過其他方式驗證狀態。
