# spell-chanting-loudspeaker 開發文檔

> 原則：簡潔。每條記錄只寫關鍵決策與狀態，不寫過程。

## 專案概要

- **引擎**：Godot 4.6.1
- **類型**：2天 Game Jam，主題「咒語」，3人團隊
- **分工**：符文系統+UI（本session負責）、戰鬥系統（另一組員）、世界觀/敘事（第三組員）
- **遊戲結構**：單一大地圖，無限生存，敵人掉落符文

## 核心機制：符文系統

### 資料流圖
- 符文串 = **DAG**（有向無環圖），DFS 遍歷執行
- 每個符文有型別化的 input/output port

| 資料型別 | 說明 |
|---------|------|
| `Energy` | 能量 |
| `HP` | 血量（可作為施放成本） |
| `DirectionVector` | 方向（單個或 List） |
| `Target` | 目標敵人 |
| `Spell` | 法術 List（Array of Dict），每個元素含 node、scene、direction 等出生資料與掛載的後修飾 |

### 執行流程
```
玩家按鍵 → 起始符文（給能量）
  → 對整個 DAG 做拓樸排序（DFS），得到確定性線性序列
  → 逐一執行序列中每個符文：播音檔 → 音檔結束 → 觸發效果 → 下一個
  → 音檔播放中再按鍵 → 施放失敗（音檔播完是平衡設計）
```
- 分岔的多條路也照拓樸順序依序執行，**音檔不重疊**

### 符文分類
| 類型 | 說明 |
|------|------|
| 起始符文 | 4個，鍵盤觸發（Q/W/E/R或類似），給一點能量 |
| 效果符文 | 火球、回血、Debuff 等，消耗 Energy 或 HP |
| 轉換符文 | 修改資料型別，例如扇形化（1個方向→3個方向的List） |
| 修飾符文 | 掛在 Spell 輸出上，例如「擊中回血」 |
| 附魔符文 | 貼附在符文上的條件修飾，例如「條件成立時傷害提高」、「多消耗能量再次觸發」 |
| 被動觸發符文 | 計時器/擊殺計數自動給能量，不需玩家按鍵 |

### 可選輸入
- 部分符文有**可選輸入 port**（多給能量效果更強）

### 連招範例

#### Target 傳遞連招（線性）
- 效果符文可以**同時輸出 `Target`**，例如：Debuff 符文對敵人上狀態後輸出該目標，後續符文接收此 `Target` 做追擊傷害
- 這意味著部分符文有多個輸出 port（如 `Spell` + `Target`）

#### 分岔連招（一輸出→多下游）
```
分岔符文（Spell + 1能量）
    ├→ 原本的路徑
    └→ 新的一點能量可以繼續接新的符文
```
兩條路照順序執行(DFS的拓樸排序)。

#### 聚合連招（多上游→一輸入）
```
起始符文（1能量）   ─┐
                    ├→ 超載閃電符文（一點必要能量，一點可選能量）
被動觸發符文（1能量）─┘
```
有時閃電符文會更強

## UI 結構

- **平時**：上半畫面=戰鬥視窗，下半畫面=符文圖編輯器
- **空白鍵**：遊戲暫停，符文圖編輯器展開至全螢幕
- 符文圖編輯器：左側畫布（自由擺放、連線），右側背包（自動排列，拖曳互轉）
- 編輯毫無限制，可隨時重組

## 戰鬥設計

- 鍵盤+滑鼠操控角色移動與技能
- **無普攻、無自動攻擊**，純技能驅動
- 敵人掉落符文

## 實作狀態

### 已完成
- **玩家移動**：WASD，速度 200，`CharacterBody2D` + `Camera2D`（跟隨）
- **分割畫面**：`SubViewportContainer`（上半）+ `Panel`（下半佔位）
- **空白鍵暫停**：`get_tree().paused`，main.gd 設 `PROCESS_MODE_ALWAYS`
- **符文系統**：完整四層架構（資料/圖/執行/UI），詳見 [符文系統.md](符文系統.md)
